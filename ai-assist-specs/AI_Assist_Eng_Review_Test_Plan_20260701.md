## Key Interactions to Verify

### Provisioning Service
- Admin publishes WISMO capability → Cognigy flow created → `LIVE` status in CXone
- Same publish repeated 100 times → identical Cognigy state each time (idempotency)
- Publish fails at Step 3 (Cognigy API 5xx) → compensating rollback → CXone record reverts → admin sees failure card
- Reconciliation job detects diverged CXone/Cognigy state → re-drives record → state converges
- Admin upgrades template version → new version marked `pending` → active conversations complete on prior version → drain window (default 5 min) → new version becomes `active`
- Admin rolls back to prior pinned version → re-publish from pinned config → Cognigy reverts to prior state

### MCP Connector
- Tool call times out (>5s) → human agent sees "Order system unavailable" card + handoff flag
- Auth token expired → one re-auth attempt → if fails, error card + admin alert
- Downstream tool returns HTTP 5xx → AI agent falls back to See-only mode → human agent sees degraded card
- Tool returns malformed response (200 OK, non-JSON body) → error card (not silent failure)

### Rules Engine
- Admin inputs NL rule → compiler produces deterministic graph → admin sees diff-style review
- Same NL rule compiled twice → two different drafts (expected) → each requires separate admin review before deploy
- Admin inputs malformed NL → compiler returns structured error → admin sees clear error message in wizard (not 500)
- Compiled graph runs same input 100 times → identical output each time (runtime determinism)
- Admin re-compiles after compiler update → produces a new draft → admin must review and publish (does not auto-replace deployed graph)

### Governance — Autonomy Mode E2E

**See-only mode:**
- AI surfaces WISMO order status card → human agent reads → no action taken
- MCP call takes >5s → human agent sees "system slow" feedback (not blank screen)
- Audit log: card render event logged with inputs, outputs, autonomy mode = See

**Suggest→approve mode:**
- AI drafts refund action → human agent sees draft card with Approve / Reject
- Human agent approves → action executes → audit log records approval
- Human agent double-clicks Approve → action executes once only (idempotency)
- Human agent navigates away before approving → draft expires → no action taken → audit log records abandonment
- MCP failure after approve → human agent sees failure card → audit log records failure

**Auto mode:**
- AI acts autonomously → human agent sees result on next turn (exception-only visibility)
- Action fails post-execution → human agent sees failure card with explanation
- Audit log: full action recorded (inputs, outputs, decision path, autonomy mode = Auto)
- Audit write fails → action is BLOCKED (fail-closed per D3) → human agent sees "action paused" card

### Governance — Audit Log
- Out-of-order write attempt → hash chain validation fails → write rejected
- WORM store unavailable during Auto mode action → action blocked (fail-closed)
- WORM store unavailable during Suggest→approve action → action proceeds; write queued for retry (fail-open)
- Auditor reads full audit log → hash chain verified end-to-end → no tampering detected
- Agent session scripted through 10 actions → all 10 appear in audit log with correct sequence

### Kill Switch
- Fleet Admin activates kill switch → new agent actions blocked (enforcement within SLA defined in Month 1)
- Kill switch activated mid-conversation → current in-flight action: behavior depends on Month 1 cache architecture
- Kill switch deactivated → actions resume

### UI Intent / Admin Surface
- Admin completes WISMO publish wizard start to finish → capability live in <30 minutes (Month 4 gate)
- Admin abandons wizard at step 3 → no partial Cognigy state created (cleanup verified)
- WISMO card renders correctly for all data types: order found, order delayed, order not found, system unavailable
- Brand token missing for tenant → fallback to default theme (no render failure)
- Schema version mismatch between runtime and renderer → clear error logged, not silent corruption

## Edge Cases

- Two admins publish the same capability simultaneously → one wins, one gets conflict error (no silent partial state)
- Admin publishes empty rule set → blocked by schema validation before publish (zero rules not allowed)
- Rapid double-publish → idempotency holds → second publish is a no-op or returns success with existing state
- Admin issues kill switch → measures actual propagation time → must meet SLA (to be defined Month 1)
- NL rule compiles to a graph that blocks ALL actions (overly restrictive) → admin sees warning in review UI
- Rules Engine compiler model updated → existing deployed graphs unaffected (compile-once model per D7)

## Critical Paths (End-to-End)

1. **Month 4 gate**: Admin (first login) → Browse WISMO template → Connect WISMO MCP → Attach shipping knowledge → Define escalation rule (NL) → Review compiled graph → Set Suggest→approve autonomy → Publish → Capability live in CXone agent workspace. Total time < 30 minutes.

2. **Month 6 Gate 1**: Customer A WISMO capability live with governance enforcement active. Human agent handles real conversation using AI assist. Audit log captures all actions. Autonomy enforcement verified (bypass attempt blocked). Compliance sign-off gate.

3. **Month 6 Gate 2**: Governance control plane passes internal compliance review. Audit trail tamper-evident (hash chain verified). Autonomy enforcement unbypassable (surface-level bypass attempt fails at runtime).

4. **Month 6 Gate 3 (conditional)**: Customer B admin completes WISMO self-serve adoption (UC-1 flow) in under 5 business days from template selection to capability live in agent workspace.

## Notes for QA Tester

- Staging tenant required: dedicated Cognigy staging tenant, provisioned Month 1 Week 1.
- All E2E tests run against staging (not production) Cognigy tenant.
- Autonomy mode E2E tests require scripted "human agent" actions (Approve/Reject clicks).
- Latency gate: P99 end-to-end round-trip must be <500ms under load before governance goes to production.
- Audit log tests require WORM-store-capable staging environment.
